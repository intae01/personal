<컴퓨팅> 서비스 : AWS 클라우드에서 앱 및 워크로드를 
               개발, 배포, 실행 및 조정

 Amazon EC2
 - 컴퓨팅 파워 제공. 컴퓨팅 용량은 전통적 온프레미스(물리서버),
   또는 가상서버에서 제공했던 기능임.
 - Amazon 머신 이미지 (AMI): 사전 인스턴스 템플릿.
   운영체제, 추가 소프트웨어 등이 포함되있음(서버 운영을 위한)

 Amazon EC2 Auto Scaling
 - 클라우드(서버)에서 실행(배포)하는 인스턴스 수를 자동으로 
   증가 및 감소시켜줌.
 - 애플리케이션 상태 및 가용성을 유지.

 Elastic Load Balancing(ELB)
 - 애플리케이션의 트래픽을 EC2 인스턴스, 컨테이너, IP주소,
   Lambda 함수등으로 자동 분산시켜 줌.

 AWS Lambda
 - 서버리스 컴퓨팅 플랫폼. 서버리스; 서버는 있는데, 걍 신경안써도 되는거
 - 자바, 파이썬, Go등 다양한 개발언어 지원.
 - 개발 코드를 작성해, 함수화 시킨다. 이를 기능별로 이름짓고 개념화
   해서 이를 배치 및 공유해 필요할때마다 사용(코드 몰라도 편하게)

 Amazon EC2 Container(ECS)
 - Docker 를 지원, 확장성이 뛰어나고 고성능인 컨테이너 관리 서비스
 - Amazon EC2 인스턴스 관리형 클러스터에서 애플리케이션을 쉽게 실행

<AWS 컴퓨팅 서비스의 주요 이점>
 (AWS클라우드에서 워크로드를 개발, 배포, 실행 및 조정할 시)

 1. 탄력성: 컴퓨팅 요구사항(서버 인스턴스 개수)변화에 따라 용량
          조정가능, 클릭 몇번으로 편하게, 몇분만에.

 2. 유연성: Windows server, Linux 등 운영체제, 소프트웨어를
          다양하게 선택가능. 이에 맞는 메모리, cpu, 인스턴스
          스토리지, 부팅 파티션 크기까지 선택 가능

 3. 보안성: Amazon EC2 Amazon Virtual Private Cloud 와
          연동하여 컴퓨팅 리소스 보안, 강력한 네트워킹 기능을 제공.
          컴퓨팅 리소스 완전 제어, 네트워크 액세스 권한 구성.
          AMI(이미지)에 보안구성 포함가능. 태그 지정시 클라우드에서
          인벤토리 추적가능.

 4. 비용절감: 규모의 경제, 비용절감효과, 사용한 컴퓨팅 용량에
            대해서만 비용을 지불함.

<스토리지> 서비스 : 데이터를 저장하는 안정적이고, 확장가능하며, 안전한 장소

 Amazon S3
 - 정적인..? 인터넷용 스토리지. 웹 규모의 컴퓨팅 작업용. 언제든지,
   용량 관계없이, 데이터 저장, 검색 및 분석(SQL, 빅데이터 분석가능)
 - 볼륨 추가와 제거가 용이하고, EBS보단 느리나 저렴한 편임.

 Amazon EBS
 - 하드 드라이브 같음. EC2에 연동됨. 파티션 생성/포맷, 운영체제
   (여러 서버에 복제됨) 고가용성, 내구성 -> 파일기반 워크로드
 - NoSQL 데이터베이스, 관계형DB, 하둡 에코시스템 등 활용가능
 - 낮은 지연시간(빠름), 한번에 용량을 정해서 사야됨.
   (추가 가능하나, S3에 비해 복잡) 인터넷 통해 외부와 연결될 수 없다.

 Amazon S3 Glacier
 - 데이터 보관 및 백업을 위한 안전 스토리지, 저렴하고 간편함.
 - 수개월/수년 보관가능. 자주 꺼내거나 수정하거나 못함..

 Amazon Storage Gateway
 - 온프레미스 소프트웨어 어플라이언스를 클라우드 기반 스토리지에 연결.
   '조직 온프레미스 IT - AWS 스토리지 인프라' 간 원활하고 안전하게 통합

 Amazon Elastic File System(EFS)
 - Amazon EC2 인스턴스용 파일 스토리지 서비스
   사용이 간편하며, 파일 시스템 생성 및 구성 가능한 인터페이스 제공.
   "여러 EC2 인스턴스간 공유가능", ACID 만족.
 - ACID(Atomicity: 원자성, Consistency: 일관성, Isolation: 고립성, Durability: 지속성)

 Amazon EC2 Instance Store
 - 임시블록 스토리지(EC2용). EC2 인스턴스 사용중에만 유지됨

<AWS 스토리지의 주요 이점>

 1. 탄력적. 요구사항 변화에 맞춰 용량을 
    조정할 수 있으며, 객체 수를 제한없이 저장가능.

 2. 유연함. 고객은 볼륨 유형, 즉 스탠다드 또는 프로비저닝된 
    온라인 트랜잭션 처리(IOPS)를 선택가능

 3. 안전함. 
    - Amazon S3 는 TLS(전계층 보안. Total Layer Security?)
      을 통한 데이터 전송을 지원, 자동 암호화. 
    - AWS(IAM)사용시, 객체 권한 관리, 데이터 액세스 제어하는 버킷
      정책 구성가능. 
    - EBS의 경우, AWS 리전간 스냅샷 복사로 거리적 확장, 마이그레이션
      복구가 용이함.

 <데이터베이스> 서비스 : 특정 애플리케이션 사용사례를 위해 특별히 구축 

 Amazon RDS
 - Amazon Aurora, PostgreSQL, MySQL, MariaDB, Oracle, SQLserver(MS-SQL) 등을 선택,
   사용할 수 있는 관리형 관계형 데이터베이스. 비용효율적, 크기조정가능
 - AWS Database Migration Service 사용시, 기존 데이터베이스를 Amazon RDS로
   마이그레이션 및 복제 가능. 관리가 쉬워짐

 Amazon Aurora
 - 고성능 상용데이터베이스의 성능과 가용성에, 오픈소스 데이터베이스의 간편성,
   비용효율성을 결합함. MySQL, PostgreSQL과 호환됨.

 Amazon Dynamo DB
 - 어떤 규모에서도 10밀리초 미만 성능. 키-값 및 문서 데이터베이스.
   빠르고 예측 가능한 성능. AWS Management Console 에서 쉽게 시작가능.
 - 가동중지, 성능저하없이 요청해 용량 조정이 가능. 리소스 사용률, 성능지표 확인가능.
 - Amazon Cloud Watch로 테이블 상태와 성능모니터링 가능.

 Amazon Elastic Cache
 - 클라우드에서 인 메모리 캐시를 손쉽게 배포, 운영 및 조정. 캐싱 서비스.
   빠른 관리형인 메모리 캐싱시스템. 정보검색 가능. 웹 어플리케이션 성능 향상.

 <데이터베이스의 이점>

 1. 손쉽게 관리 가능. 고객이 프로비저닝이나, 패치, 백업등 관리 걱정 안해도됨.
    AWS가 지속적 모니터링. 고객은 애플리케이션 개발 집중이 가능.(거진 다 서버리스..)

 2. 가용성과 내구성 우수. 고객이 다중 AZ 데이터베이스 인스턴스를 프로비저닝 하는 경우
    Amazon RDS는 다른 AZ의 대기 인스턴스에 동기적으로 데이터를 복제함.

 3. 확장성이 매우 뛰어남. 대부분 가동을 중단하지 않고, 몇번의 클릭이나 API호출 만으로
    데이터베이스의 컴퓨팅 및 스토리지 리소스 조정 가능. 

 <보안, 자격증명 및 규정 준수> 서비스 : 암호화, 액세스 관리 및 규제되는 워크로드 보안.

 AWS IAM 
 - AWS 서비스와 리소스에 대한 사용자 액세스 제어. IAM 사용시 기업이 AWS 에서 사용자를
   생성하고 관리할 수 있음. 
 - 또한, AWS 외부에서 관리되는 사용자에게 AWS 리소스 액세스 권한 부여 가능.

 AWS Directory Service
 - 고객이 AWS 리소스를 기존 온프레미스 Microsoft Active Directory와 연결, 
   혹 AWS 클라우드에서 새로운 독립 실행형 디렉토리 설정시 사용가능 관리형서비스

 AWS Cloud HSM
 - 고객이 AWS 클라우드에서 전용 HSM 어플라이언스를 사용함으로써, 데이터
   보안에 대한 기업, 계약 및 규정 준수 요구사항을 충족하도록 해줌.
 - 암호화 키 및 HSM이 수행하는 암호화 작업제어 가능.

 <보안, 자격증명 및 규정 준수 이점>

 1. AWS 는 인프라 스트럭처에서 수십개의 규정 준수 프로그램을 관리해,
    고객의 규정 준수 요구사항을 충족함. 즉, 고객의 규정 준수 일부가
    이미 충족되어 있음.

 2. 고객은 자체시설을 관리할 필요가 없음. 보안 비용이 절감됨.

 3. 고객의 AWS 사용량에 따라 보안 범위도 확장됨. 모든 비즈니스 규모에서
    안전 유지 설계

 4. 고객 개인정보보호를 하도록 강력한 보안조치. 모든 데이터가 
    고도로 보안된 AWS 데이터 센터에 저장됨.

 <관리 및 거버넌스> 서비스 : 클라우드 환경에 대한 완벽한 제어

 Amazon Cloud Watch
 - AWS에서 실행하는 리소스와 애플리케이션 모니터링.
   CPU/네트워크사용률(시스템지표), 페이지로드/쿼리실행시간(사용자지표) 포함
 - EC2 인스턴스, DynamoDB테이블, RDS등 AWS 리소스와 고객 어플리케이션,
   로그 등도 모니터링 가능함.

 AWS Cloud Trail
 - 고객이 AWS 계정의 거버넌스, 규정준수, 운영감사, 위험감사를
   수행하도록 해줌. AWS 인프라 스트럭서에서의 작업과 관련있는
   계정활동 로깅. 지속적 모니터링. 데이터 보존.

 AWS Systems Manager
 - AWS 기반 인프라스트럭처 가시적 확인 및 제어. 통합사용자 인터페이스 통해
   AWS 운영 데이터 확인, 운영 작업 자동화 가능. 
 - Amazon EC2 인스턴스, S3 버킷, RDS 인스턴스등 AWS 리소스를
   애플리케이션별로 그룹화, 모니터링, 조치를 취할 수 있음.

 AWS Cloud Formation -> 텍스트 파일 사용해 프로비저닝! 코드형 인프라 프로비저닝. 

 - 간단한 텍스트 파일 사용. 자동화되고 안전한 방식으로 모든 리전,
   계정에 걸쳐 애플리케이션에 필요한 리소스를 모델링 및 프로비저닝
   할 수 있음. 공통언어를 제공.

 AWS Auto Scaling
 - 애플리케이션을 모니터링해 용량을 자동으로 조정.
   안정적이고 예측가능한 성능을 유지함. 강력한 인터페이스 제공.
 - EC2 인스턴스 및 스팟, Amazon ECS(도커쓰는거), DynamoDB테이블 및
   인덱스, Aurora 복제본 리소스에 대한 규모 조정계획 수립.

 <관리 및 거버넌스 이점>

 1. 안정성. 일관된 제어를 제공.

 2. 사용편의성. 프로그래밍 방식으로 프로비저닝.

 3. 모니터링. 지표를 수집 및 추적.

 4. 관리. 거버넌스 및 규정준수를 충족.
 
 <네트워킹> 서비스 : 클라우드 인프라 스트럭처를 격리하고 요청처리, 용량조정

 Amazon VPC
 - 클라우드에서 가상 네트워크를 구축. 
   고객은 격리된 프라이빗 네트워크 영역을 프로비저닝하고 리소스를 배포 가능. 
 - VPC는 Elastic Load Balancing을 연동하면 강력한 네트워킹과 보안 가능.
 
 AWS Direct Connect
 - 고객 데이터센터와 AWS 간의 프라이빗 연결. 표준 1기가 비트 또는 10기가 비트
   이더넷 광섬유 케이블 사용.
 
 Amazon Route 53
 - 고도로 확장가능한 DNS 서비스. 최종사용자를 인터넷 어플리케이션으로 라우팅

 <네트워킹의 이점>

 1. 탄력성. 요청 처리를 조정 ex) Route 53은 고객 개입없이도
    대규모 쿼리 볼륨을 처리하게끔 자동확장이 가능.

 2. 사용편의성. 단일보기에서 연결을 확인, 연결 및 가상 인터페이스를 
    포괄적으로 관리가능

 3. 보안성. 물리적 및 가상 프라이빗 네트워크를 연결. 인스턴스 레벨과
    서브넷 레벨에서 모두 인바운드 및 아웃바운드 필터링 사용(VPC고급기능)

 4. 비용절감. 비용 효율적. VPC 자체는 생성 및 사용에 별도 요금없음.
    (하드웨어 VPN 연결로 기업 데이터 센터에 별도로 연결시, 시간당 요금은 부과됨)

 <6R 마이그레이션 전략>
 
 1. 리호스팅(Rehosting)
 - 애플리케이션 -> 클라우드 ' 리프트 앤 시프트'
 - 각 서버는 EC2 인스턴스로 마이그레이션. 네트워크 구성은 VPC, 서브넷, 보안그룹,
   인터넷 및 VPN 게이트웨이 등을 사용해 재생성
 - AWS 에서만 호스팅되는 온프레미스 네트워크를 기능적으로 그냥 재현함..
   (큰 교체가 적은 방식이라 보면 됨)
 - 종량제 모델의 이점을 누리고, 필요시 빠른 추가 가능. 비용절감 가능
 
 2. 리플랫폼(Replatform)
 - '리프트 앤 시프트' 및 '수정'
 - 애플리케이션 핵심 아키텍처가 변경되지 않는다는 점에서 리호스팅과 유사
 - 기존 old한 데이터베이스를 Amazon RDS로 바꾸거나(관리형 솔루션, AWS상에서 관리하기 쉬우라고.) 
   완전 관리형 플랫폼으로 애플리케이션을 마이그레이션 하는등.. 안전하고 복원력 뛰어남.
   인프라 최적화 가능.
 
 3. 리팩터링(Refactor) : '현대화'
 - 구형 온프레미스 아키텍처를 클라우드 네이티브로 '재설계(현대화)'
 - 최종기능은 비슷해보일지언정, 내용은(기술적으로) 다름.

 4. 사용중지(Retire)
 - 유용하지 않은 애플리케이션 종료. 
   지출, 관리 및 보안 부담이 감소.

 5. 유지/재검토(Retain/Revisit)
 - 특정 애플리케이션의 경우 온프레미스 상태로 유지함.

 6. 재구매(Repurchase)
 - AWS에서 지원하는 더 나은 SaaS(Software as a Service)로 교체(또 산다.)

 - 거버넌스 관점은 IT 거버넌스 모범 사례를 식별 및 구현,
   기술로 "비즈니스 프로세스"를 지원.

 - 운영 관점은 비즈니스 "이해당사자"의 요구 사항을 충족,
   IT "워크로드를 운영" 및 복구하는 데 중점을 둠.

 - 비즈니스 관점은 비즈니스 "전략"과 IT 전략을 분리하는 모델.
   IT 전략을 통합하는 비즈니스 모델로 "전환"하는 데 도움
